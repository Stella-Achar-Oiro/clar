CLAR — How it works

End-to-end technical walkthrough · Last updated 2026-04-27

What CLAR does

CLAR accepts a medical report (PDF or plain text), runs it through a 5-agent LangGraph pipeline, and returns structured plain-English findings with urgency flags and doctor questions. PII is stripped before any LLM call. Nothing is persisted to disk.

Supported report types: lab, radiology, discharge summary, pathology.

POST /api/upload — request lifecycle

Entry point: app/api/routes/upload.py. The route is async but the pipeline is CPU+IO-bound, so it is offloaded to a thread pool to keep the event loop free (prevents Cloud Run liveness probe timeouts).

Client sends multipart/form-data with field file (.pdf or .txt, max 10 MB).

Route validates extension (.pdf / .txt) and file size. Returns 415 or 413 on failure.

File is written to a temp file; run_pipeline(tmp_path) is called via loop.run_in_executor(None, ...). Temp file is deleted in the finally block whether or not the pipeline succeeds.

Pipeline result (CLARState) is inspected. If error == "deid_failed" → 422. Any other error → 500.

Findings are assembled into ReportResult. A Verdict is computed (urgent > watch > normal) based on finding urgency counts.

Session is saved in the in-memory store keyed by report_id (UUID v4) with a 30-minute TTL.

Prometheus metrics are incremented: REQUESTS_TOTAL, PIPELINE_DURATION, DEID_ENTITIES_TOTAL.

200 response with full ReportResult JSON returned to client.

# upload.py — key snippet
loop = asyncio.get_event_loop()
state = await loop.run_in_executor(None, run_pipeline, tmp_path)

if state.get("error") == "deid_failed":
    raise HTTPException(status_code=422, detail="Could not safely process this document.")

The graph has one conditional edge: after DeIDAgent, deid_router() routes to error_node if de-identification failed, or to explain_agent if clean. All other edges are fixed sequential (explain → flag → advisor → END). 

# pipeline.py — graph wiring
graph.set_entry_point("deid_agent")
graph.add_conditional_edges(
    "deid_agent", deid_router,
    {"explain_agent": "explain_agent", "error_node": "error_node"},
)
graph.add_edge("explain_agent", "flag_agent")
graph.add_edge("flag_agent", "advisor_agent")
graph.add_edge("advisor_agent", END)

Shared state — every agent receives the full CLARState TypedDict and returns a new dict merging its changes with {**state, ...}: 

class CLARState(TypedDict):
    raw_text:      str          # from ExtractAgent
    deid_text:     str          # from DeIDAgent
    report_type:   str          # lab | radiology | pathology | discharge
    findings:      list[dict]   # from ExplainAgent
    explanations:  list[dict]   # from ExplainAgent (+ plain_explanation)
    flagged:       list[dict]   # from FlagAgent (+ urgency, urgency_reason)
    questions:     list[str]    # from AdvisorAgent (exactly 5)
    deid_entities: list[dict]   # entity type + count for audit log
    error:         str | None

Agent 1 — ExtractAgent

File: app/agents/extract_agent.py · app/services/extractor.py

Runs before the LangGraph graph (needs the file path). Two responsibilities: extract raw text and detect report type.

PDF extraction

pdfplumber opens the file. For each page, tables are extracted first (pipe-separated rows preserve column structure). Free-text lines are then appended, deduplicated against table content, so narrative sections in discharge/radiology reports aren't lost. 

for table in page.extract_tables():
    for row in table:
        cells = [c.replace("\n", " ").strip() if c else "" for c in row]
        parts.append(" | ".join(cells))

text = page.extract_text()
if text:
    table_content = " ".join(parts)
    for line in text.splitlines():
        if line.strip() and line.strip() not in table_content:
            parts.append(line)

Report type detection

Keyword dict ordered by specificity (discharge → pathology → radiology → lab). First match wins. The ordering prevents generic keywords from shadowing specific ones — e.g. "findings:" was removed from radiology because it appears in every report type. 

_REPORT_TYPE_KEYWORDS = {
    "discharge": ["discharge summary", "discharge diagnosis", ...],
    "pathology": ["pathology report", "histopathology", "biopsy", ...],
    "radiology": ["radiology", "x-ray", "mri", "ct scan", ...],
    "lab":       ["cbc", "full blood count", "haemoglobin", ...],
}

Agent 2 — DeIDAgent (safety gate)

File: app/agents/deid_agent.py · app/services/deid.py

Critical safety gate. If de-identification fails or produces empty text, the pipeline routes to error_node and the upload returns 422. PII never reaches the LLM under any circumstances.

Uses Microsoft Presidio (AnalyzerEngine + AnonymizerEngine). Two custom recognisers are added on top of the built-in set:

RecogniserPatternReplacement
MRNMRN[\s:-]{0,2}\d{6,10}[MEDICAL_ID]
US SSN\b\d{3}-\d{2}-\d{4}\b[MEDICAL_ID]

Built-in entities detected: PERSON → [PATIENT], DATE_TIME → [DATE], LOCATION → [ADDRESS], PHONE_NUMBER → [PHONE], EMAIL_ADDRESS → [EMAIL], UK_NHS / MEDICAL_LICENSE → [MEDICAL_ID].

Every removed entity is logged as a structured warning (type, character offsets, confidence score) — audit trail without logging the actual PII value.

# deid_agent.py — conditional router
def deid_router(state: CLARState) -> str:
    if state.get("error") == "deid_failed":
        return "error_node"
    return "explain_agent"

Agent 3 — ExplainAgent

File: app/agents/explain_agent.py · app/prompts/explain.py · app/prompts/explain_examples.py

Sends deid_text + report_type to Claude. Returns structured JSON: one object per finding with name, value, unit, reference_range, plain_explanation (max 3 sentences), confidence.

Prompting strategy

Few-shot prompting with 4 in-context examples — one per report type (CBC lab, chest X-ray radiology, pneumonia discharge, breast biopsy pathology). Each example is a full user→assistant pair showing the exact JSON schema the model must produce. This teaches schema consistency without relying on JSON mode alone.

Temperature 0.1 for consistency. Max tokens 8192 to handle reports with many findings.

# explain.py — few-shot structure
FEW_SHOT_EXAMPLES = [
    {"role": "user",      "content": "Analyse this lab report:\n..."},
    {"role": "assistant", "content": CBC_RESPONSE},   # from explain_examples.py
    {"role": "user",      "content": "Analyse this radiology report:\n..."},
    {"role": "assistant", "content": RADIOLOGY_RESPONSE},
    # ... discharge + pathology pairs
]

explain_examples.py is a separate file (146 lines) that holds all response string constants. This keeps explain.py under the 150-line project hard cap.

Agent 4 — FlagAgent

File: app/agents/flag_agent.py · app/prompts/flag.py

Classifies each finding as urgent watch normal. Uses a rules-first, LLM-fallback architecture.

Rules engine (no LLM cost)

Two patterns checked in order:

  1. Qualitative positive: if value matches /positive/i and reference range matches /negative|not detected/i → urgent. Catches pathology receptor status without an LLM call.
  2. Numeric range: regex extracts value and low/high bounds. Deviation > 50% of normal span → urgent; any deviation → watch.
def classify_numeric(value, low, high):
    normal_span = high - low
    if low <= value <= high: return "normal", "Within range."
    deficit = (low - value) / normal_span  # if below
    urgency = "urgent" if deficit > 0.50 else "watch"
    return urgency, f"Value is below the normal range ({value} vs {low}–{high})."

LLM fallback

When neither pattern matches (qualitative findings like tumour grade, imaging descriptors, discharge diagnoses), a chain-of-thought prompt is sent to Claude with the flag prompt. The system prompt covers four domains: imaging severity, discharge findings, pathology findings, and qualitative lab results. Temperature 0.1, max 1000 tokens.

Agent 5 — AdvisorAgent

File: app/agents/advisor_agent.py · app/prompts/advisor.py

Generates exactly 5 questions the patient should ask their doctor. Questions are specific to the actual flagged values — not generic health advice. Temperature 0.4 (higher than other agents) for natural variation in phrasing.

The user message explicitly names the report type twice so the model tailors question style appropriately (lab values vs imaging findings vs pathology results).

LLM service

File: app/services/llm.py

Single shared Anthropic client with a 30-second timeout and exponential backoff retry (up to 3 attempts, 2s→4s→8s) on transient errors (429, 500, 502, 503, 529, timeout). 

@retry(
    retry=retry_if_exception_type(_RETRYABLE),
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=8),
    reraise=True,
)

After each call, input and output token counts are pushed to Prometheus (LLM_TOKENS counter, labelled by agent name and direction).

JSON repair

If the LLM response fails json.loads() (e.g. truncated at max_tokens), _repair_truncated_json() attempts recovery: regex-extracts complete finding objects before the truncation point. For non-finding shapes it returns safe empty defaults rather than raising.

De-identification service

File: app/services/deid.py

deidentify(text) returns a 3-tuple: (anonymised_text, entity_list, deid_failed). It never raises — on any exception it returns ("", [], True) so the pipeline can abort cleanly via the conditional edge.

Session store

File: app/services/session.py

In-memory dict keyed by report_id (UUID v4). TTL 30 minutes. Stores: findings, questions, report_type. No database. No disk writes. minScale=1 on Cloud Run prevents cold starts that would clear the store mid-session.

CI / CD

ci.yml — every PR

ruff lint → mypy type-check → pytest >80% coverage → Next.js build → docker build

cd.yml — merge to main

All CI steps → GCP Workload Identity auth (no long-lived service account keys) → docker build + push to Artifact Registry (--platform linux/amd64) → gcloud run deploy → curl /health smoke test

NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY is a Docker build arg — baked into the static JS bundle at build time, not a runtime env var.

TDD discipline

Every agent and service was test-written first (red → implement → green). Tests live in tests/unit/ and tests/integration/. CI enforces >80% coverage; PRs cannot merge without green tests.

Error handling strategy

All HTTP errors are raised as HTTPException in route handlers. Agents catch their own exceptions, log them, and return {**state, "error": "..."} so the pipeline continues to END and the route handler inspects the error field. This means no agent can crash the server — the worst outcome is a 500 with an actionable message.

The LLM client has a 30s timeout + 3-attempt retry with exponential backoff, so transient API blips are handled transparently.
